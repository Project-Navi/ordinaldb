# OrdinalDB 0.3.0 API, Async, and Streaming Spec

Status: design target after the 0.2.0 storage and adapter parity release.

## Goal

0.3.0 should make OrdinalDB's public API explicit about concurrency,
nonblocking integration, cancellation, and result streaming. The release should
turn the 0.2.0 redb-backed storage foundation into a coherent runtime contract.

## Concurrency Contract

- Readers use immutable vector generations.
- Writers are serialized per adapter store.
- A writer builds and verifies a new vector generation before publishing it.
- Publishing is a short redb transaction that updates active generation,
  mappings, documents, metadata, tombstones, and audit records.
- Existing readers keep using the generation they opened.
- Failed writes leave the previous generation active.
- Open fails closed on active-generation digest, table-count, or mapping
  mismatch.

## Rust API

Add explicit handles:

- `Database`
- `Collection`
- `ReadSnapshot`
- `WriteBatch`
- `SearchQuery`
- `SearchPage`
- `SearchStream`

Read APIs:

- `collection.snapshot()`
- `snapshot.search(query)`
- `snapshot.search_page(query, page_token)`
- `snapshot.get(ids)`
- `snapshot.filter_to_allowlist(filter)`

Write APIs:

- `collection.write_batch()`
- `batch.add(...)`
- `batch.upsert(...)`
- `batch.delete(...)`
- `batch.commit()`
- `batch.abort()`

## Async API

Rust async should not pretend redb or core search are native async. Use bounded
blocking pools and clear cancellation points:

- async open/load may use blocking sections.
- async search submits blocking vector work and can cancel before dispatch.
- async writes can cancel before the publish transaction.
- once a short publish transaction starts, it either commits or fails closed.

Python adapters should use `asyncio.to_thread` or framework-native executors for
0.3.0 async methods. PyO3 search/write calls should release the GIL around heavy
Rust work.

## Streaming

0.2.0 can stream only after a full top-k search completes. 0.3.0 should add real
paged search:

- core search iterator or page API.
- stable page tokens tied to a read snapshot and query fingerprint.
- backpressure-aware batch size.
- cancellation between pages.
- deterministic ordering for ties within a snapshot.
- clear behavior when `k`, filter allowlists, or tombstones change between
  snapshots.

Initial API shape:

```rust
let snapshot = collection.snapshot()?;
let mut stream = snapshot.search_stream(query).limit(100).page_size(16)?;
while let Some(page) = stream.next_page()? {
    // hydrate page records from the same snapshot.
}
```

Python shape:

```python
async for record in store.asearch_stream(query, k=100, page_size=16):
    ...
```

## Framework Adapters

- LangChain: async add/search/delete/get; streaming result iterator only if the
  framework surface supports it without pretending to stream tokens.
- LlamaIndex: async query and storage context compatibility.
- Haystack: async document store/retriever parity where Haystack exposes async.
- Agno: async methods may delegate to the common async facade, but should expose
  cancellation and no-embedder errors consistently.

## Required Tests

- concurrent readers see stable snapshots during a write.
- failed write leaves old generation readable.
- cancellation before dispatch does not mutate state.
- cancellation during publish resolves to committed or failed-closed state.
- streaming pages cover the same IDs as non-streamed top-k for margin-safe
  fixtures.
- page tokens reject use against a different snapshot/query.
- Python event loop remains responsive during async adapter calls.

## Non-Goals

- No distributed writer coordination.
- No network server protocol.
- No full-text or hybrid streaming until those query types exist.
- No claim of native async redb IO.
