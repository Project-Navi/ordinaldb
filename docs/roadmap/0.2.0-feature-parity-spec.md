# OrdinalDB 0.2.0 Feature-Parity Spec

Status: draft for the 0.2.0 PR train.

## Release Goal

OrdinalDB 0.2.0 should be credible as an embeddable integration-layer vector
DB for local agent runtimes and enterprise-search prototypes. The core remains
OrdVec-backed, ordinal, and vector-only; adapters own framework text, metadata,
string IDs, stable handles, and integration semantics.

## Source Posture

- The fork baseline is MIT-licensed provenance. Its framework adapter
  semantics, tests, and persistence integrity patterns may be adapted with
  local attribution when copied directly. See `docs/provenance.md` and
  `THIRD_PARTY.md`.
- Non-ordinal vector engines, learned codebooks, rotation pipelines, and
  normalized relevance-score claims are permanent non-goals, not deferred
  implementation work.
- The adapter-store architecture follows a redb serving pattern: verified
  hot-path artifacts, manifest/table-count checks, fail-closed open, and
  SQLite kept out of the runtime source-of-truth path.

## Comparison Matrix

| Area | Provenance / ecosystem signal | LanceDB-style expectation | OrdinalDB 0.2.0 decision |
| --- | --- | --- | --- |
| Core vector engine | Historical upstream included non-ordinal engines | Multiple metrics and index types | Keep OrdVec-only core and document score semantics |
| Persistence | JSON sidecars with integrity tests | Table/storage abstraction with indexes and filtering | Add redb adapter control plane; keep `.odb` vector-only |
| Stable IDs | Stable ID map and delete semantics | Stable row handles | Preserve invariants in redb verified open |
| LangChain | InMemoryVectorStore-like behavior | Framework-native vector store | Implement sync parity: upsert, filters, dump/load aliases, get/delete |
| LlamaIndex | Node fidelity and StorageContext paths | Framework-native persistence | Implement node fidelity, filters, get/delete nodes, persist/load |
| Haystack | InMemoryDocumentStore parity | Document store plus retriever | Implement filter DSL, duplicate policy, utility methods, save/load aliases |
| Agno | LanceDb-shaped lifecycle | Vector DB lifecycle and content hashes | Implement lifecycle, content-hash upsert, delete-by APIs where compatible |
| Full text / hybrid | Not core | FTS and hybrid search are first-class LanceDB features | Reject for 0.2.0; document vector-only adapter scope |
| Async / streaming | Upstream has async adapter conveniences | Async query builders and materialization controls | Defer public API to 0.3.0; do not overclaim |
| CI / release | Broad adapter and release tests | Package confidence across integrations | Add all-extras adapter job, sdist gate, deny gates, redb corruption tests |

## PR Lanes

### 1. Storage

Add `ordinaldb-adapter-store`, a redb-backed adapter/control-plane crate used
by CLI and Python. It owns:

- `adapter.redb` manifest and table counts.
- stable string ID to `u64` mappings.
- `u64` to vector slot mappings.
- documents and metadata JSON.
- generation checkpoints bound to active `.odb` manifest digest, vector count,
  dim, and bit width.
- reserved pending-batch, tombstone, GC, and audit tables.

Acceptance:

- `cargo test -p ordinaldb-adapter-store`
- `cargo test -p ordinaldb-cli`
- Python common adapter tests prove save/load through redb.
- `cargo tree -p ordinaldb` shows no redb or SQLite dependency in the core
  crate.

### 2. Adapter Semantics

Adapt framework behavior where it fits the OrdinalDB boundary.

- LangChain: existing-ID adds upsert, in-batch duplicate keep-last, callable
  filters receive `Document`, missing delete is a no-op, dump/load aliases.
- LlamaIndex: full node serialization, `get_nodes`, `delete_nodes`, local
  persist/load paths, richer filters, and node/filter allowlist intersection.
- Haystack: use native filter helper when available, richer duplicate policies,
  save/load aliases, and metadata helper methods. Keep blob/sparse fields out
  until OrdinalDB stores the required payloads reliably.
- Agno: lifecycle methods, content-hash upsert, delete-by ID/name/metadata or
  content ID, supported search type reporting, no-embedder text-search errors.

Keep rejected in 0.2.0: MMR, sparse, text-search, hybrid, normalized relevance
scores, and full-vector return APIs unless OrdinalDB stores the required data.

### 3. Release and CI

- Add an all-extras Python adapter job that installs `ordinaldb[adapters]`.
- Keep a base-wheel job for import-boundary and missing-extra tests.
- Gate `maturin sdist`.
- Expand `cargo deny` to advisories, bans, licenses, and sources.
- Add Python ARM64 wheel smoke if hosted runners are available.
- Keep existing Rust ARM64 and Pi runtime jobs.

### 4. Documentation

- Document redb storage, legacy JSON imports, and vector-only core boundary.
- Add adapter parity matrix and unsupported-mode matrix.
- Add attribution notes when baseline code or tests are copied directly.
- Keep score wording as "OrdinalDB similarity scores", not cosine.

## 0.2.0 Non-Goals

- No SQLite runtime dependency.
- No non-ordinal vector engines.
- No full-text or hybrid search engine.
- No public async/streaming contract beyond existing framework compatibility
  shims.
- No claim of power-loss durability beyond verified temp writes and fail-closed
  open.
